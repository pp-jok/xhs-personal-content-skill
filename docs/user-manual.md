# 使用手册

## 1. 这个 Skill 是什么

本 Skill 用于辅助单个小红书创作者长期运营账号。你可以把它当成一个“账号运营陪跑助手”：它会帮你记录账号定位、拆解对标内容、沉淀你的偏好、生成选题和草稿，并根据你的反馈持续调整。

你不需要直接编辑数据文件。正常使用时，只要在 Codex 里用自然语言告诉它你的账号信息、对标帖子和反馈即可。

## 2. 使用原则

这个 Skill 的重点不是一次性生成一篇文案，而是让它越来越理解你的账号。

建议持续输入：

- 你的账号定位和目标用户。
- 你喜欢的对标帖子。
- 你不喜欢的表达。
- 你希望保留的标题、封面和脚本风格。
- 已发布内容的表现和复盘。

## 3. 固定触发词

建议每次都以这个前缀开始：

```text
小红书账号运营，...
```

常用句式：

```text
小红书账号运营，初始化这个项目的账号工作区
小红书账号运营，更新我的账号档案
小红书账号运营，添加这个对标账号
小红书账号运营，分析这个截图，加入对标库
小红书账号运营，把这个链接加入素材库
小红书账号运营，根据我的账号生成选题
小红书账号运营，根据这个选题生成草稿
小红书账号运营，创建发布任务
小红书账号运营，跑一下真实样本验证
小红书账号运营，记录这篇草稿的质量评价
小红书账号运营，生成本周质量报告
小红书账号运营，记录我的反馈：这个标题太 AI，这篇先不要这样写；先作为候选偏好让我确认
小红书账号运营，复盘这篇内容
```

## 4. 第一次使用

在新项目中说：

```text
小红书账号运营，初始化这个项目的账号工作区
```

它会先引导你补账号底座。你可以直接复制下面这段填：

```text
小红书账号运营，更新我的账号档案：

账号名：
账号定位：
目标用户：
我想吸引的人：
我不想吸引的人：
内容风格：
我喜欢的标题风格：
我讨厌的表达：
可借鉴方向：
不可做的内容：
近期目标：
```

如果你还没想清楚，可以先填 3 到 5 项。Skill 应该先保存可用信息，再告诉你还缺哪些关键内容。

## 5. 添加对标账号

可以提供账号链接、截图、简介或自然语言描述：

```text
小红书账号运营，添加这个对标账号。
账号名：
链接：
为什么值得学习：
我想学的点：
我不想学的点：
```

对标账号是持续输入，不是一次性配置。后续可随时补充。

## 6. 添加对标帖子

可以发送截图、链接或复制正文：

```text
小红书账号运营，分析这个截图，加入对标库。
我喜欢它的点是：
我不想学的点是：
这个帖子适合我借鉴的地方是：
```

Codex 只应转录用户提供或截图中可见的信息，不应编造缺失数据。

如果只提供链接，Skill 会先把它放进素材收件箱，并记录你想学什么。它不会批量抓取平台，也不会绕过登录、验证码或访问限制。内容不完整时，它会告诉你缺标题、正文、封面、互动数据还是评论，并请你补截图或复制可见内容。

链接输入示例：

```text
小红书账号运营，把这个链接加入素材库：
链接：
我觉得好在：
我想重点学：标题 / 选题 / 封面 / 脚本结构
```

添加后，Skill 应该用普通话术告诉你：

- 这篇已加入素材库。
- 它最值得学的地方是什么。
- 哪些地方不适合你直接学。
- 下一步应该补充什么判断。

当链接或截图内容已经进入素材库后，你可以继续说：

```text
小红书账号运营，拆解这篇素材是否值得对标
```

Skill 应该区分“看得见的事实”和“基于事实的推断”，并明确告诉你哪些地方还不确定。只有你确认值得学习后，再把它提升为正式对标内容。

## 7. 生成选题和草稿

当账号档案、对标帖子和标签已有一定积累后，可以说：

```text
小红书账号运营，根据我的账号生成 10 个选题
```

或：

```text
小红书账号运营，根据这个选题生成草稿，要求标题、封面文案、视频逐字稿和简单分镜都给出来
```

生成时应优先参考本地项目中的账号档案、标签、规则卡片和历史反馈。

## 8. 记录反馈

这是调教质量的关键。用户可以直接说：

```text
小红书账号运营，记录我的反馈：这个标题太 AI，以后少用夸张承诺
```

或：

```text
小红书账号运营，记录我的反馈：这个封面标题可以，保留“反常识 + 明确对象”的写法
```

反馈应被沉淀为标签、规则卡片备注或验证反馈，而不是只停留在当前对话。

规则会有生命周期：

- 候选：从对标拆解或反馈中提炼出来，但还没确认。
- 已确认：你明确说这条适合当前账号。
- 测试中：已经用于选题、草稿或发布任务。
- 已验证：发布结果支持这条规则。
- 已拒绝：不适合当前账号。
- 已废弃：被更具体或更有效的规则替代。

你可以直接说：

```text
小红书账号运营，确认这条规则适合我
小红书账号运营，这条规则验证成功
小红书账号运营，废弃这条规则，因为有更具体的写法
```

每条规则都应尽量保留证据：来自哪篇对标、哪段可见内容、对应的推断是什么。

## 9. 让它更懂你的方法

每次看到不满意的输出，不要只说“不好”，尽量补一句原因：

```text
小红书账号运营，记录我的反馈：这个标题太 AI，因为它像模板句，不像真人经验。
```

```text
小红书账号运营，记录我的反馈：这个选题不适合我，因为我的账号不想做争议型内容。
```

```text
小红书账号运营，记录我的反馈：这个封面标题可以，因为对象明确，而且有结果感。
```

好的反馈会变成后续规则。对标帖子和反馈越多，生成结果越贴合你的账号。

## 10. 跑真实样本验证

在新项目中说：

```text
小红书账号运营，跑一下真实样本验证
```

正常情况下，Skill 应直接告诉你验证结果，不需要把命令和文件路径展示出来。

你应该重点看：

- 目前样本够不够。
- 是否已经沉淀出有效规则。
- 选题是否值得继续扩展。
- 草稿是否需要人工大改。
- 下一轮最该补哪类对标帖。

## 11. 记录草稿质量评价

当 Skill 生成草稿后，不要只看“有没有内容”，要记录它需要改多少、哪里不像你的账号。

你可以直接说：

```text
小红书账号运营，记录这篇草稿的质量评价：
账号适配 4 分
可发布 3 分
标题 2 分，因为太像模板
封面 4 分
脚本 3 分，需要更口语
需要修改 2 次，不需要大改
```

这类评价会影响后续规则：哪些表达应该保留，哪些标题、封面或脚本模式应该减少。

## 12. 生成本周质量报告

连续记录几篇草稿的质量评价后，可以说：

```text
小红书账号运营，生成本周质量报告
```

你应该重点看：

- 首轮通过率有没有提高。
- 平均修改次数有没有下降。
- 哪些规则真的有效。
- 哪些规则经常导致标题或脚本被重写。
- 下一周应该补哪类对标帖。

这一步是判断 Skill 是否越用越贴合账号的关键，不是统计生成数量。

## 13. 什么时候需要技术细节

普通使用不需要看技术细节。只有当你想排查问题或准备开发时，再说：

```text
小红书账号运营，显示技术细节
```

这时 Codex 可以展示文件路径、命令、数据文件和验证结果。

## 14. 当前限制

当前版本更适合做账号运营知识库、对标分析、规则沉淀和本地验证。正式内容质量主要依赖 Codex 对话读取你的本地账号资料后生成，并通过持续反馈迭代。

本项目只处理用户主动提供的单个链接和用户可见内容，不做平台级爬虫、批量抓取、账号监控或访问限制绕过；不会自动发布，也不会接入外部模型 API。
