# 使用手册

## 1. 这个 Skill 是什么

本 Skill 用于辅助单个小红书创作者长期运营账号。它把账号档案、对标内容、用户偏好、规则卡片、选题、草稿、发布任务和复盘记录保存为本地 JSON / Markdown 文件。

用户不需要直接编辑 JSON。推荐方式是在 Codex 中用自然语言触发，Codex 根据 Skill 说明读写项目本地工作区。

## 2. 推荐工作区

在任意新 Codex 项目中，推荐使用：

```text
.xhs-personal-content-skill/real-sample/
```

全局 Skill 目录只放引擎和说明，不保存测试者的真实账号资料。

## 3. 固定触发词

建议每次都以这个前缀开始：

```text
小红书账号运营，...
```

常用句式：

```text
小红书账号运营，初始化这个项目的账号工作区
小红书账号运营，更新我的账号档案
小红书账号运营，添加这个对标账号
小红书账号运营，分析这个截图，加入对标库
小红书账号运营，根据我的账号生成选题
小红书账号运营，根据这个选题生成草稿
小红书账号运营，创建发布任务
小红书账号运营，跑一下真实样本验证
小红书账号运营，记录我的反馈：这个标题太 AI，以后不要这样写
小红书账号运营，复盘这篇内容
```

## 4. 第一次使用

在新项目中说：

```text
小红书账号运营，初始化这个项目的账号工作区
```

然后补充账号档案：

```text
小红书账号运营，更新我的账号档案：

账号名：
账号定位：
目标用户：
我想吸引的人：
我不想吸引的人：
内容风格：
我喜欢的标题风格：
我讨厌的表达：
可借鉴方向：
不可做的内容：
近期目标：
```

如果信息不完整，Codex 应最多问 3 个关键问题，不应要求用户一次填完所有字段。

## 5. 添加对标账号

可以提供账号链接、截图、简介或自然语言描述：

```text
小红书账号运营，添加这个对标账号。
账号名：
链接：
为什么值得学习：
我想学的点：
我不想学的点：
```

对标账号是持续输入，不是一次性配置。后续可随时补充。

## 6. 添加对标帖子

可以发送截图、链接或复制正文：

```text
小红书账号运营，分析这个截图，加入对标库。
我喜欢它的点是：
我不想学的点是：
这个帖子适合我借鉴的地方是：
```

Codex 只应转录用户提供或截图中可见的信息，不应编造缺失数据。

## 7. 生成选题和草稿

当账号档案、对标帖子和标签已有一定积累后，可以说：

```text
小红书账号运营，根据我的账号生成 10 个选题
```

或：

```text
小红书账号运营，根据这个选题生成草稿，要求标题、封面文案、视频逐字稿和简单分镜都给出来
```

生成时应优先参考本地项目中的账号档案、标签、规则卡片和历史反馈。

## 8. 记录反馈

这是调教质量的关键。用户可以直接说：

```text
小红书账号运营，记录我的反馈：这个标题太 AI，以后少用夸张承诺
```

或：

```text
小红书账号运营，记录我的反馈：这个封面标题可以，保留“反常识 + 明确对象”的写法
```

反馈应被沉淀为标签、规则卡片备注或验证反馈，而不是只停留在当前对话。

## 9. 跑真实样本验证

在新项目中说：

```text
小红书账号运营，跑一下真实样本验证
```

Codex 应从全局 Skill 引擎目录运行：

```bash
cd ~/.codex/skills/xhs-personal-content-skill
python3 -m app.cli validate-real-sample --workspace /absolute/path/to/project/.xhs-personal-content-skill/real-sample
```

验证报告输出到：

```text
.xhs-personal-content-skill/real-sample/reports/validation_report.md
.xhs-personal-content-skill/real-sample/reports/human_review_form.md
```

## 10. 当前限制

当前 CLI 的批量生成使用 mock 服务，适合验证数据结构和流程闭环。正式内容质量应通过 Codex 对话读取本地数据后生成，并通过持续反馈迭代。

本项目不会自动抓取平台内容、不会自动发布、不会接入外部模型 API。
