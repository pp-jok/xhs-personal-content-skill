# 公开测试指南

## 1. 测试目标

本测试不是评估一次性文案生成能力，而是评估 Skill 是否能围绕单个创作者账号持续学习：

- 能否保存真实账号档案。
- 能否持续录入对标账号和帖子。
- 能否把对标样本拆成可复用规则。
- 能否根据账号档案、标签和规则生成更贴合个人账号的选题和草稿。
- 能否根据人工反馈更新偏好。

## 2. 测试前准备

测试者需要准备：

- 一个测试用 Codex 项目。
- 1 份账号档案。
- 1 到 3 个对标账号。
- 3 到 10 篇对标帖子截图或文字。
- 至少 5 条个人偏好，例如喜欢的标题、讨厌的表达、不能碰的选题。

不需要提供账号密码，不需要授权自动抓取，不需要授权自动发布。

## 3. 安装方式

克隆仓库后执行：

```bash
mkdir -p ~/.codex/skills
cp -R xhs-personal-content-skill ~/.codex/skills/xhs-personal-content-skill
```

重启 Codex。

## 4. 测试流程

### 第一步：初始化工作区

```text
小红书账号运营，初始化这个项目的账号工作区
```

预期结果：

- 创建或确认 `.xhs-personal-content-skill/real-sample/`。
- 说明后续账号数据会保存在当前项目内。

### 第二步：补充账号档案

```text
小红书账号运营，更新我的账号档案：
账号名：
定位：
目标用户：
内容风格：
禁用表达：
近期目标：
```

预期结果：

- 写入或更新 `creator_profile.json`。
- 总结已保存的信息和仍缺失的信息。

### 第三步：添加对标内容

```text
小红书账号运营，分析这个截图，加入对标库。
我喜欢它的点是：
我不想学的点是：
```

预期结果：

- 写入或更新 `benchmark_account.json`。
- 写入或更新 `benchmark_post.json`。
- 只记录用户提供或截图中可见的信息。

### 第四步：记录偏好

```text
小红书账号运营，记录我的反馈：这个标题太 AI，以后不要这样写
```

预期结果：

- 偏好进入标签、规则卡片备注或验证反馈。
- 后续生成时能避开类似问题。

### 第五步：运行验证

```text
小红书账号运营，跑一下真实样本验证
```

预期结果：

- 生成 `reports/validation_report.md`。
- 生成 `reports/human_review_form.md`。
- 报告列出输入数量、规则卡片数量、选题数量、草稿数量、发布任务数量和每一步状态。

## 5. 评价标准

请按 1 到 5 分评价：

```text
1 = 不可用
2 = 需要大改
3 = 可参考
4 = 小改可用
5 = 可直接使用
```

评价维度：

- 选题是否适合账号。
- 标题是否可用。
- 封面标题是否可用。
- 视频逐字稿是否能录。
- 是否符合账号风格。
- 是否不像 AI。
- 是否有结果感。
- 是否接地气。
- 是否比通用内容生成更贴合个人账号。
- 是否愿意连续使用 2 周。

## 6. 反馈格式

建议测试者提交以下反馈：

```text
测试账号定位：
输入了多少篇对标帖子：
最有用的功能：
最不好用的步骤：
生成内容最大问题：
是否愿意继续用 2 周：
建议下一步优先改什么：
```

## 7. 已知限制

- 当前没有 UI。
- 当前不做真实平台抓取。
- 当前不做自动发布。
- 当前 CLI 生成链路使用 mock 服务，主要验证数据流。
- 高质量内容生成依赖 Codex 对话读取本地项目数据后完成。
