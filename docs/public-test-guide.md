# 公开测试指南

## 1. 测试目标

本测试不是评估一次性文案生成能力，而是评估 Skill 是否能围绕单个创作者账号持续学习：

- 能否保存真实账号档案。
- 能否持续录入对标账号和帖子。
- 能否把对标样本拆成可复用规则。
- 能否根据账号档案、标签和规则生成更贴合个人账号的选题和草稿。
- 能否根据人工反馈更新偏好。

## 2. 测试前准备

测试者需要准备：

- 一个测试用 Codex 项目。
- 1 份账号档案。
- 1 到 3 个对标账号。
- 3 到 10 篇对标帖子截图或文字。
- 可选：1 到 3 个用户主动提供的小红书链接，用于测试素材收件箱。
- 至少 5 条个人偏好，例如喜欢的标题、讨厌的表达、不能碰的选题。
- 至少 3 条草稿质量评价，用于测试质量周报是否能指出修改成本和规则效果。

不需要提供账号密码，不需要授权批量抓取，不需要授权自动发布。

## 3. 安装方式

克隆仓库后执行：

```bash
mkdir -p ~/.codex/skills
cp -R xhs-personal-content-skill ~/.codex/skills/xhs-personal-content-skill
```

重启 Codex。

## 4. 测试流程

### 第一步：初始化工作区

```text
小红书账号运营，初始化这个项目的账号工作区
```

预期结果：

- 用普通话术说明账号工作区已准备好。
- 不主动暴露文件路径、命令或内部数据结构。
- 引导测试者先补账号定位、目标用户、内容风格、禁用表达和近期目标。

### 第二步：补充账号档案

```text
小红书账号运营，更新我的账号档案：
账号名：
定位：
目标用户：
内容风格：
禁用表达：
近期目标：
```

预期结果：

- 说明哪些账号信息已经保存。
- 指出还缺哪些会影响生成质量的信息。
- 最多追问 3 个问题，不要求测试者一次填完所有信息。

### 第三步：添加对标内容

```text
小红书账号运营，分析这个截图，加入对标库。
我喜欢它的点是：
我不想学的点是：
```

预期结果：

- 说明这篇内容已加入素材库。
- 提炼可借鉴点和不可直接照搬的风险。
- 只记录用户提供或截图中可见的信息。
- 用一个简单问题引导测试者说明最想学哪一点。

### 第三步补充：添加单个链接

```text
小红书账号运营，把这个链接加入素材库。
链接：
我觉得好在：
我想重点学：
```

预期结果：

- 链接进入素材收件箱。
- 同一链接重复提交不会重复生成素材。
- 如果内容无法直接获取，应明确说明缺少标题、正文、封面、互动数据或评论。
- 不绕过登录、验证码、风控或访问限制。
- 引导测试者补截图、复制正文或说明喜欢原因。

### 第三步补充：拆解素材并确认是否对标

```text
小红书账号运营，拆解这篇素材是否值得对标
```

预期结果：

- 明确区分看得见的事实和基于事实的判断。
- 明确列出还不确定的信息。
- 说明哪些点适合当前账号学习，哪些点不适合照搬。
- 不把点赞、收藏、评论解释成确定的爆款原因。
- 询问测试者是否确认提升为正式对标内容。

### 第四步：记录偏好

```text
小红书账号运营，记录我的反馈：这个标题太 AI，这篇先不要这样写；先作为候选偏好让我确认
```

预期结果：

- 明确复述这条反馈。
- 说明它会先作为候选偏好，等待用户决定后才影响后续标题、封面、选题或脚本。
- 不要求测试者理解标签、规则卡片或数据文件。

### 第五步：运行验证

```text
小红书账号运营，跑一下真实样本验证
```

预期结果：

- 用普通话术说明验证是否完成。
- 总结样本数量是否够用、规则是否有效、选题和草稿是否值得人工评估。
- 默认不展示命令和文件路径。
- 如果测试者说“显示技术细节”，再展示报告位置和命令。

### 第六步：记录质量评价并生成周报

```text
小红书账号运营，记录这篇草稿的质量评价：
账号适配 4 分，可发布 3 分，标题 2 分，脚本 3 分，需要改 2 次
```

然后继续说：

```text
小红书账号运营，生成本周质量报告
```

预期结果：

- 说明内容是否越来越少改。
- 指出有效规则和表现差的规则。
- 指出标题、封面或脚本里被重复否定的模式。
- 给出下一轮最该补充的样本类型。
- 不用生成数量代替质量判断。

## 5. 评价标准

请按 1 到 5 分评价：

```text
1 = 不可用
2 = 需要大改
3 = 可参考
4 = 小改可用
5 = 可直接使用
```

评价维度：

- 引导是否顺畅。
- 是否少暴露系统层面的文件、命令和数据结构。
- 是否能告诉我下一步该补什么。
- 规则是否能追溯到具体证据。
- 规则状态是否清楚，例如候选、已确认、测试中、已验证、已拒绝或已废弃。
- 选题是否适合账号。
- 标题是否可用。
- 封面标题是否可用。
- 视频逐字稿是否能录。
- 是否符合账号风格。
- 是否不像 AI。
- 是否有结果感。
- 是否接地气。
- 是否比通用内容生成更贴合个人账号。
- 是否愿意连续使用 2 周。
- 是否能看出修改成本正在下降。
- 是否能指出哪些规则真正有效，哪些规则需要调整。

## 6. 反馈格式

建议测试者提交以下反馈：

```text
测试账号定位：
输入了多少篇对标帖子：
引导是否顺畅：
有没有暴露太多技术细节：
最有用的功能：
最不好用的步骤：
生成内容最大问题：
质量周报是否有用：
是否愿意继续用 2 周：
建议下一步优先改什么：
```

## 7. 已知限制

- 当前没有 UI。
- 当前不做平台级爬虫、批量抓取、账号监控或访问限制绕过。
- 当前不做自动发布。
- 高质量内容生成依赖 Codex 对话读取本地项目数据后完成。
- 技术细节默认隐藏；需要排查问题时可以让 Codex 显示技术细节。
